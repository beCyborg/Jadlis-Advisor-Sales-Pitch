# Follow-Through Phase: Steps 4–8

Source: April Dunford, *Sales Pitch* (2023)

---

## Phase Prerequisites

Follow-Through begins **only** after the "Right?" gate at the end of Step 3 returns explicit prospect agreement. If that gate has not been passed, do not proceed to Step 4.

The Follow-Through phase shifts topic from the market (what the Setup is about) to the solution (what the Follow-Through is about). Everything in Steps 4–8 is justified by the criteria the prospect agreed to in Step 3. Every value theme, every proof point, every ask is traceable back to that agreement.

---

## Step 4: Introduction

### Purpose

Orient the prospect to what they are about to see. Name the market category. Move on. This step takes under 60 seconds.

### What to do

Introduce the company and product using the market category identified during positioning. One slide or one sentence. The prospect should know what type of thing they are evaluating before the demo starts — not during it.

For companies with multiple products: show the full portfolio briefly (family diagram), then specify which product is the focus. Do not explain the whole portfolio in depth. The prospect is here for one thing.

For single-product companies: state market category and the segment it is built for.

### Diagram types by product architecture

| Diagram Type | When to Use | What It Shows |
|--------------|-------------|---------------|
| Family diagram | Multi-product companies | How products fit together; which product is the focus of this call |
| Platform diagram | Platform products | Full range of functionality; which components are optional |
| Marketecture diagram | Technical / infrastructure products | How the product fits into the buyer's existing tech stack |

**Marketecture diagram rules:** Must be simple enough to explain in one pass. If explaining the diagram takes more than 90 seconds, it is too complex. The prospect should immediately see where your product connects to their existing environment.

### Common error

Over-explaining the category or introducing a new category name that requires its own setup. Use the category name the prospect already recognizes. If your category name requires definition, the Introduction step will expand to fill the time that should go to the Value Demo.

### Transition

As soon as the prospect knows what type of product they are seeing, move to Step 5. There is no value in extending the Introduction.

## Step 5: Differentiated Value Demo

### Purpose

Demonstrate how the product delivers the value the prospect agreed to in Step 3. This is the bulk of the pitch — the step where the deal is won or lost. All other steps are scaffolding for this one.

### Core organizing principle

Organize the demo around **value themes**, not product menus, user workflows, or feature lists.

A value theme is a cluster of features and outcomes that together deliver one specific dimension of differentiated value. Maximum 3 themes. If you identify more than 3, select the 3 that most directly address the criteria from Step 3 for this specific prospect.

**Why max 3:** Prospects cannot hold more than 3 things in working memory across a first call. More than 3 themes produces a feature tour, not a value demonstration. The prospect leaves remembering capabilities, not value.

### Demo script structure

```
"We deliver a combination of [Value Theme A], [Value Theme B], and [Value Theme C] for our customers."

"Let me show you [Value Theme A] first — here are the specific capabilities that deliver that."
[Demo or slide for Theme A]

"Now [Value Theme B] — here's how we handle that."
[Demo or slide for Theme B]

"And [Value Theme C] — here's what that looks like in practice."
[Demo or slide for Theme C]
```

Each theme introduction should name the theme explicitly. Do not let the prospect infer the theme from the features. Name it, then show it, then move on.

### Three delivery formats

| Format | When to Use | Structure |
|--------|------------|-----------|
| Slides + Demo | Product shows well AND there is non-demo value to communicate (pricing model, services, integrations) | Slide names value theme → live demo walks through features for that theme → slide restates the theme and outcome |
| Demo Only | UX or experience IS the primary value; slides slow the narrative | Demo is scripted by value theme; each section of the demo is explicitly introduced by naming the theme |
| Slides Only | Product does not demo well (complex infrastructure, data pipelines, security, AI models); or no live product yet | Screenshots, architecture diagrams, example implementations, benchmark data; organized by value theme |

**When to use Slides + Demo:** Default format for most B2B products. Use it unless there is a specific reason to use one of the other formats.

**When to use Demo Only:** The prospect must *feel* the product to understand the value (e.g., consumer-grade UX, design tools, developer tools where the DX is the differentiator). Even in Demo Only, the value themes must be scripted in advance — improvised demo tours default to menu structure.

**When to use Slides Only:** The product does not convey value from live interaction alone (e.g., data warehouses, ML pipelines, security monitoring). Screenshots with annotations showing specific outcomes are more persuasive than watching data load.

### What to include

Include only features and capabilities that:
1. Deliver one of the three value themes
2. Are differentiating — not available in the alternatives from Step 2 in the same way

Features that provide necessary assurance (security certifications, SLA commitments, compliance attestations) but are not differentiating belong in Step 7 (Objections), not the Value Demo.

### What to exclude

| Excluded Content | Why |
|-----------------|-----|
| Login, account setup, onboarding flows | Not differentiating. Wastes time on commoditized functionality. |
| Admin and settings panels | Not what the champion cares about. Relevant only for the IT stakeholder in a later meeting. |
| Features that don't connect to a value theme | No frame for the prospect to evaluate them. Adds to cognitive load without adding to the value case. |
| Every capability the product has | Creates a feature tour. The prospect evaluates completeness, not differentiated fit. |
| Customized deep-dive demo | Save for second meeting after deeper discovery. First call demo is standardized around value themes. |

### Worked examples

**Help Scout (customer support software)**

Three value themes:
1. Easy-to-use inbox experience — simple enough that support agents actually use it
2. Advanced features you won't outgrow — as team scales, no platform migration required
3. Great customer experience — customers get responses that feel personal, not templated

Demo organized around these three themes, not around Help Scout's navigation menu (Inbox, Docs, Reports, etc.).

**Funnel (marketing data platform)**

Four value themes (note: exception to max-3 rule, justified by prospect type):
1. Brings all marketing data together from any source
2. Stores in a central hub that marketing owns (not IT dependency)
3. Auto-prepares data for analysis (no manual cleaning)
4. Makes data available wherever needed (downstream BI tools, ad platforms, etc.)

Demo shows each theme in 2–3 clicks, with the prospect's own source types named explicitly.

**Gearset (Salesforce DevOps)**

Three value themes:
1. DevOps done the right way — uses standard engineering tools (Git, CI/CD), not Salesforce-specific workarounds
2. Deepest understanding of Salesforce DevOps — validated across 2,000+ real deployments
3. Broad and deep Salesforce-specific capabilities — addresses the full range of org management problems

Demo does not follow the Gearset product menu (Deployments, Version Control, Testing, Backup). It follows these three themes.

### Key principle

Features are **evidence** for value claims, not the main attraction. The prospect should leave the Value Demo remembering the three themes, not a list of features. If they can recite the value themes but not the specific features, the demo succeeded. If they can recite features but not the themes, the demo failed.

## Step 6: Proof

### Purpose

Provide external validation for the value claims made in Step 5. The prospect has seen the product — now they need to believe the outcomes are real.

Rule: **You saying something about your product is not proof. A customer saying it is proof.** First-party claims have zero credibility as proof. All proof must be external.

### 8-Tier Proof Hierarchy

Use the highest available tier for each value claim. Do not present all tiers — select the best match.

| Tier | Type | Description | When to Use |
|------|------|-------------|-------------|
| 1 | Matching case study | Same industry, same company size, same problem, quantified outcome | Always lead with this if available |
| 2 | Third-party verification | Independent audit, lab benchmark, analyst validation | When credibility of the claim itself is in question |
| 3 | Certifications | SOC 2, ISO 27001, HIPAA, FedRAMP, etc. | Regulated industries; security and compliance objections |
| 4 | Customer-validated statistics | Metrics confirmed by customers ("30% reduction in X") | When outcomes are measurable and customers have confirmed them |
| 5 | Customer quotes | Named customer, title, company; specific outcome claim | When Tier 1–4 not available for this value claim |
| 6 | Analyst quotes / review platform ratings | Gartner, Forrester, G2, Capterra | Enterprise buyers who trust analyst opinions |
| 7 | Third-party research | Industry survey, academic study supporting the general claim | Supporting context when first-party proof is weak |
| 8 | Awards | Industry awards, best-of lists | Use only when nothing else is available |

### Proof matching matrix

Match proof to the prospect across two independent dimensions:

**Dimension 1 — Industry and company match**

| Proof Match Level | Description | Impact |
|-------------------|-------------|--------|
| Exact match | Same industry, same size, same situation | Maximum credibility transfer |
| Category match | Same industry, different size | Strong; note the size difference explicitly |
| Problem match | Different industry, same problem pattern | Moderate; frame around problem similarity |
| No match | Unrelated industry and situation | Weakest; use only if nothing better exists |

**Dimension 2 — Role match**

The case study or quote source should be in the same role as the prospect. A CFO evaluating ROI wants financial outcomes cited by another CFO, not an engineer. A developer wants a technical case study from another developer team.

| Prospect Role | Match Proof Source To |
|---------------|-----------------------|
| Economic buyer (CFO, VP) | ROI outcome, cost reduction, revenue impact |
| Technical evaluator (CTO, architect) | Implementation story, integration complexity, performance data |
| End user champion (VP Marketing, Director of Sales) | Workflow improvement, time savings, adoption rates |
| Procurement / legal | Certifications, compliance attestations, contract terms from comparable deals |

### Timing: weaving vs. dedicated section

Two valid approaches:

**Weave into Step 5:** Mention the case study immediately after demonstrating the relevant value theme. "That's the automated compliance tracking — and [Company X] in your industry saw a 40% reduction in audit prep time using exactly this." Keeps the narrative fluid.

**Dedicated Step 6 section:** Show a proof slide or case study summary after the full demo. Works for enterprise buyers who expect formal validation before advancing. Also works when you have a strong Tier 1 case study that deserves its own moment.

Both approaches are valid. Do not use both in the same pitch unless they serve different value themes.

### Case study structure (Tier 1)

A proof-quality case study contains:
1. **Who:** Company name, industry, size (enough for the prospect to see themselves)
2. **Before:** The specific problem they had — using the same framing as your Insight from Step 1
3. **How:** What they used (feature set, implementation approach) — brief
4. **After:** Quantified outcome, confirmed by the customer (not vendor-estimated)

If any of these four elements is missing, the case study drops in effective tier. A case study with no quantified outcome is Tier 5 (customer quote), not Tier 1.

## Step 7: Objections (Conditional)

### Purpose

Proactively neutralize predictable concerns that would prevent the deal from advancing — even if the prospect has not raised them.

### Decision rule: include or skip

This step is **conditional**. Apply the following test:

| Condition | Action |
|-----------|--------|
| A common objection exists AND you have a strong response | Include this step |
| A common objection exists AND you do NOT have a strong response | Skip. Do not proactively surface a concern you cannot resolve. |
| No common unspoken objections in this segment | Skip. No Step 7 needed. |
| All concerns were already raised and handled during Steps 1–6 | Skip. Do not repeat. |

**Critical rule:** Never proactively raise an objection you cannot handle. Surfacing a concern without resolution creates doubt without relief. This actively damages the deal.

### Common objection categories and response patterns

**Deployment / implementation complexity**

```
"One thing teams often ask at this point is how long implementation takes.
For a company your size, typical onboarding is [X weeks / days].
We have a dedicated implementation team — your account manager works with
[specific role] at your organization and handles [specific tasks].
[Company Y], a [comparable company], was live in [timeframe].
I can connect you with them if it would help to hear their experience directly."
```

Requirements for this response to work:
- Specific timeline (not "it depends")
- Named support resource (not "our team")
- Reference customer for the deployment path (optional but strong)

**Pricing / budget**

```
"Pricing is usually a question at this stage. We price based on [model: seats /
usage / contract value]. To give you a concrete sense: companies like yours
typically see packages in the range of [range]. That said, pricing is customized
based on [variables]. The best next step is a scoping call so I can put together
an accurate proposal."
```

Requirements: Never give a single price point without context. Show a range tied to company size or usage. Offer to scope accurately rather than guess.

**User adoption**

```
"Teams sometimes worry about getting [end users] to actually use the product.
We've built adoption support directly into our process: [training format,
onboarding support, champion enablement]. Adoption rate across our customers
is [stat if available]. [Customer quote from similar end user if available]."
```

**Integration with existing stack**

```
"We have [N] pre-built integrations with the tools you're likely using —
including [specific tools relevant to prospect]. For anything outside that list,
we have an API. Typical integration setup is [timeframe]. Our technical team
can review your stack in detail in a follow-up."
```

**Security and compliance**

```
"Security and compliance come up a lot in [prospect's industry]. We're
[SOC 2 Type II / ISO 27001 / HIPAA certified — whatever applies].
We also work with [regulated companies in same vertical]. Happy to set up
a dedicated call with our security team and yours if that's the right next step."
```

**Partial adoption / "we only need part of this"**

```
"Most customers start with [core module] and expand from there. You don't need
to deploy the full platform to get value from [the part they care about].
We can scope a focused implementation around [their priority] and add [other
capabilities] when the time is right."
```

### Format

State the concern directly, then resolve it. Do not hedge. Do not apologize for the concern. Do not over-explain. One concern, one resolution.

Template: "Teams in your situation often ask about [concern]. Here's how that works: [specific, non-hedged resolution]."

### Transition to Step 8

Objection handling leads naturally into the Ask. The transition happens when the last objection is resolved. A common bridge: "Now that we've walked through how this would work for your team — here's what I'd recommend as a next step."

## Step 8: The Ask

### Purpose

End the pitch with a specific recommendation, not a question. Guide the buyer to the appropriate next step.

**Rule:** Never end with "let us know if you have questions" or "what are your thoughts?" Those are not asks. They transfer responsibility to the prospect for a process they understand less well than you do. You know what the logical next step is. State it.

### Decision tree by prospect state

| Prospect Signal | Recommended Ask |
|----------------|----------------|
| Enthusiastic, asking implementation questions | POC or trial with defined scope and timeline |
| Interested but says "I need to loop in [name]" | Offer targeted presentation for the specific stakeholder(s) |
| Technical evaluation required ("our IT team would need to look at this") | Propose technical review meeting; offer to connect your technical team |
| Asking about contract or pricing specifics | Scoping call to produce an accurate proposal |
| Ready to purchase (procurement language active) | Request specific next step: proposal sign-off, procurement intro, contract review |
| Interested but no urgency signal | Schedule follow-up with specific date, specific agenda, specific owner |
| Criteria mismatch, not a good fit | Acknowledge the mismatch; suggest an alternative if one exists; close cleanly |

### POC / trial ask template

```
"Based on what we've discussed, I'd recommend a proof of concept.

Here's what that would look like:
- We'd need [specific people] on your side involved
- Typical scope for a company like yours is [specific deliverable or use case]
- Timeline is usually [X weeks]
- We'd start with [first concrete step — e.g., a kickoff call, data access, specific config]

Does that make sense as a next step?"
```

Requirements for this ask to work:
- **Specific stakeholders:** Who from their team needs to be involved (not "your team")
- **Specific scope:** What gets evaluated (not "the product")
- **Specific timeline:** A number (not "a few weeks")
- **Specific first step:** What happens immediately after they say yes

Without these specifics, the "yes" produces inaction. The prospect doesn't know how to proceed.

### Stakeholder presentation ask template

```
"It sounds like [name / role] will need to be part of this decision.
I'd like to set up a targeted session for them — focused specifically on
[what matters to that stakeholder].
I can tailor it to their questions.

Could you introduce me, or would it be easier if we scheduled a three-way call?"
```

### Graceful disqualification template

```
"Based on what you've described, [criterion they care about most] is really the
priority — and that's not where we focus.

I'd be wasting your time showing you more of the product. [Alternative, if known]
might be a better fit for what you've described.

Happy to stay in touch if your priorities shift."
```

Graceful disqualification is a **positive outcome**. Fast disqualification saves selling time and prevents a bad-fit customer from entering the sales process, where they consume resources and eventually churn or produce a bad reference.

### Rules for The Ask

1. Make one specific ask, not multiple options. Offering three choices produces analysis paralysis and diffuses momentum.
2. Make the ask easy to say yes to — low commitment, clear scope, obvious first step.
3. Do not ask "what do you want to do next?" You are the expert on your sales process. Recommend the right next step.
4. Be direct about timelines and requirements. Vague asks produce vague commitments.
5. If the prospect is not a fit, say so clearly and end cleanly. Do not soft-pedal the mismatch and schedule a follow-up out of politeness.

### First call goal clarification

The goal of a first call is not to close every prospect. It is:

- Get every **good-fit** prospect past this stage to a concrete next step
- Make clear to every **bad-fit** prospect that they should look elsewhere

Both outcomes are wins. The failure mode is good-fit prospects leaving with no next step scheduled, or bad-fit prospects entering a pipeline they will eventually stall. The Ask step determines which outcome occurs.

## Follow-Through Phase Anti-Patterns

| Anti-Pattern | Signal | Correct Action |
|--------------|--------|---------------|
| Introduction expands into a second pitch | Introduction slide takes more than 60 seconds to explain | Cut to one sentence + market category. Nothing else. |
| Value Demo organized by product menu | Demo follows UI navigation structure (File, Edit, View) | Reorganize by 3 value themes derived from Step 3 criteria |
| Non-differentiating features included in demo | Demo covers compliance badges, login, settings, admin | Move to objection step or cut entirely |
| Proof from wrong segment | Banking case study shown to a healthcare prospect | Match case study to prospect industry or use Tier 5+ instead |
| Objection raised without a resolution | "Implementation can be complex, but..." with no concrete answer | Do not raise this objection. Remove from Step 7. |
| Multiple asks presented | "You could do a trial, or a pilot, or a proposal, or..." | Pick one. Make the single best recommendation. |
| Ending with "any questions?" | Final words are "let us know if you have questions" | Replace with explicit next step recommendation |
| Customized deep-dive demo on first call | Demo shows prospect-specific data or custom configuration | Save customized demo for second call post-discovery |

## Step Sequence Summary

| Step | Time Allocation | Gate |
|------|-----------------|------|
| 4 — Introduction | < 60 seconds | None. Move immediately to Step 5. |
| 5 — Value Demo | Majority of call time | All 3 themes covered |
| 6 — Proof | 5–10 minutes or woven into Step 5 | Highest available tier used for each claim |
| 7 — Objections | 3–7 minutes (if included) | Only included when good response exists |
| 8 — The Ask | 2–3 minutes | Specific next step agreed or disqualification explicit |

For time allocation across the full call, see [storyboard-overview.md](storyboard-overview.md) timing guidelines.
